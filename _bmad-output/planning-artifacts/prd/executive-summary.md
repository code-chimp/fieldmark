# Executive Summary

FieldMark is a Construction Compliance & Inspection Management System (CCIMS) implemented three times — in .NET (Razor Pages), Django (Templates), and Go (Fiber) — against one infrastructure-owned PostgreSQL domain schema. Each stack uses HTMX for interactivity and AG Grid as a deliberate JavaScript island for data-dense views. There are no client-side state stores, no duplicated business rules, and no SPA orchestration in any of the three implementations.

The product is a teaching artifact, not a market entrant. Its primary user is a skeptical mid-to-senior engineer evaluating whether HTMX is viable at enterprise scale; its in-app personas — Project Manager, Compliance Officer, Site Supervisor, Executive — exist to make workflows realistic enough that the architectural argument survives scrutiny. The domain (project lifecycle, inspection workflow, violation management, server-evaluated compliance scoring, immutable audit trail) is rich enough to require state machines, RBAC, and rule evaluation, but bounded so the architectural message is never lost behind feature volume.

The thesis: for applications whose interactions are fundamentally request-response, the SPA default trades durable architectural complexity for marginal interactivity gains, and a server-authoritative HTMX implementation is a strictly better trade — provided it can match SPA-equivalent interaction smoothness. FieldMark is built to make that "provided" falsifiable in front of an audience that has lived with the costs being argued away.

## What Makes This Special

**Three parallel stacks against one shared domain.** Many HTMX demos are single-stack toys. FieldMark holds routes, HTMX target IDs, AG Grid endpoint contracts, audit action strings, and domain method names identical across .NET, Django, and Go. The framework becomes the variable; backend authority becomes the constant. Story-level parity is enforced at every boundary — a story is not done until all three stacks pass it.

**Pragmatic, not dogmatic.** Introductory HTMX teaching emphasizes hypermedia-first patterns — the right focus for showing what makes HTMX distinctive. Less attention has been paid to rich-island integration with mature third-party controls like AG Grid or Kendo Grid. The gap is reasonable but it leaves teams that depend on those controls without a worked example, which can lead to the mistaken conclusion that adopting HTMX requires giving up tooling they rely on. FieldMark integrates AG Grid (server-side row model only) as an explicit, scoped JS island. The grid never owns business logic; row selection fires HTMX requests for detail panels; the server still decides everything. JSON endpoints supporting the grid aren't pure HATEOAS, and that's fine — they're valid, prevalent, and useful. The concession to a mature third-party control is the point: the architecture survives realistic constraints, not just rhetorical ones.

**Server authority as defense, not just architecture.** Every state transition, every validation, every "is this button rendered" decision is server-side. The backend can never trust data sent from the front end — it is trivial for any user to alter requests before they leave the browser using DevTools, an intercepting proxy like Burp Suite or mitmproxy, or a replay tool like Postman. Client-side validation is a UX courtesy, never a security control. FieldMark's authority model is a security posture as much as an architectural one. Audit entries are written in the same transaction as the change they record. Compliance scores recompute server-side on every relevant write. The client requests HTML.

**SPA-equivalent interaction smoothness as a UX target.** "Delight" here is narrowly defined: an inspection completes, a violation resolves, a corrective action is approved — and the relevant tile, panel, and audit row update inline, in one round trip, with no full-page reload and no client-side orchestration. If a skeptic in the audience can't tell from the interaction feel that this isn't a SPA, the thesis lands.

**Backend-stack hyperfocus as a productivity argument.** The genesis of FieldMark is over a decade of Angular/React experience layered on top of non-Node backends (.NET, Java, Python, Rails) and the compounding cost of staying current with both halves. Server-driven HTMX lets a team invest deeply in one stack instead of being half-good at two. FieldMark's three-stack symmetry exists to make that point stack-independent.
